# Research: Подтверждение перед отправкой (007)

## 1. Нативный `window.confirm` vs модальный диалог в UI

**Decision**: Использовать модальный диалог в интерфейсе панели (HeroUI `Modal` / аналог из уже подключённой библиотеки), а не `window.confirm`.

**Rationale**:

- Спека требует сводку (получатели, вложения, фрагмент текста); нативный диалог не позволяет отформатированный многострочный контент и слабо предсказуем по стилю и доступности между браузерами.
- Конституция (принцип V) предполагает осмысленную разметку, фокус и читаемость — модальный паттерн на компонентах дизайн-системы ближе к этому.

**Alternatives considered**:

- `window.confirm` — минимум кода, но не выполняет FR-004 (богатая сводка), хуже для клавиатуры/скринридеров в составе сложного текста.
- Отдельная страница «review» — избыточно для одного шага и усложняет сохранение `File[]` при навигации.

## 2. Где формировать сводку получателей

**Decision**: На клиенте, из уже загруженного списка `chats` и состояния `mode` / `selectedIds`: для `all` — текст вида «All chats (N)» с N = `chats.length`; для `subset` — «Selected: k chat(s)» и при необходимости краткий список имён (с усечением).

**Rationale**: Спека допускает снимок на момент подтверждения; отдельный API за текущим числом чатов не требуется и не добавляет защиты (гонки между операторами вне scope).

**Alternatives considered**:

- `GET /api/chats` перед отправкой — лишняя задержка и дублирование данных, уже переданных на страницу.

## 3. Превью длинного текста в модалке

**Decision**: Показывать первые N символов (например 500–800) плюс явная строка «… (truncated, full length: L characters)» при `content.length > N`.

**Rationale**: Соответствует acceptance scenario 4 в спеке; избегает раздувания DOM и сохраняет читаемость.

**Alternatives considered**:

- Полный текст — допустим для средних сообщений, но 2048 символов в модалке перегружает UI; лучше единое правило усечения.

## 4. Двойной клик и параллельные отправки

**Decision**: Сохранить и расширить текущий паттерн: `pending === true` отключает кнопки «Отправить» на форме и действия подтверждения в модалке до завершения `fetch`; модалка закрывается после успешного ответа или остаётся открытой при ошибке — по продуктовому усмотрению (рекомендация: закрыть при старте отправки и показывать ошибку на основной странице, как сейчас).

**Rationale**: FR-005 уже частично покрыт `pending`; модалка не должна позволить второй confirm пока идёт запрос.
