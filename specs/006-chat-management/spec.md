# Feature Specification: Управление чатами и автоматическая регистрация

**Feature Branch**: `006-chat-management`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: «управление чатами» — как сделать так, чтобы при добавлении бота в чат он автоматически попадал в учёт (в список чатов для админа).

## Clarifications

### Session 2026-04-01

- Q: Нужна ли в продукте регистрация webhook у Telegram из интерфейса? → A: Да — в **настройках бота** администратор должен иметь возможность **зарегистрировать webhook** (привязать endpoint этого развёртывания к боту у Telegram), чтобы входящие обновления доходили без обходных ручных шагов вне панели.
- Q: Как задаётся URL webhook при регистрации? → A: **Вариант A** — администратор **всегда вручную** вводит **полный** URL webhook (ни автоподстановка по умолчанию, ни обязательное автоиз конфигурации без редактирования).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Новый чат появляется в учёте сам (Priority: P1)

Как администратор, я хочу, чтобы каждый чат, куда добавили бота (или откуда бот начал получать события), автоматически появлялся в списке чатов панели, чтобы не вносить чаты вручную и не пропускать аудиторию для рассылок.

**Why this priority**: Без автоматической регистрации список чатов неполный и рассылки и обзор некорректны.

**Independent Test**: Добавить бота в новый групповой чат (или инициировать первое событие от чата), затем открыть список чатов в панели и убедиться, что чат появился без ручного ввода.

**Acceptance Scenarios**:

1. **Given** бот только что добавлен в новый чат, **When** система получает от Telegram информацию, достаточную для идентификации этого чата, **Then** чат регистрируется и становится виден администратору в списке чатов в течение разумного времени.
2. **Given** чат уже был в учёте, **When** снова происходит активность из того же чата, **Then** запись не дублируется: администратор по-прежнему видит один чат с тем же логическим идентификатором.

---

### User Story 2 — Понятная карточка чата для админа (Priority: P2)

Как администратор, я хочу видеть в списке осмысленное название (или понятный запасной вариант) и признак активности, чтобы отличать чаты и доверять списку при выборе получателей рассылки.

**Why this priority**: Регистрация без отображаемых атрибутов снижает ценность управления.

**Independent Test**: Открыть список чатов и проверить, что у зарегистрированного чата отображаются ожидаемые поля (название/заголовок, идентификатор, признак активности — в рамках продукта).

**Acceptance Scenarios**:

1. **Given** чат зарегистрирован, **When** администратор просматривает список, **Then** он видит стабильный идентификатор чата и человекочитаемое имя или согласованный запасной текст, если имя недоступно.

---

### User Story 3 — Отражение выхода бота из чата (Priority: P3)

Как администратор, я хочу, чтобы при удалении бота из чата или потере доступа это отражалось в учёте (например, «неактивен»), чтобы не считать такой чат надёжным получателем рассылки.

**Why this priority**: Снижает ошибки при таргетинге; менее критично, чем первичная регистрация.

**Independent Test**: Удалить бота из чата (или смоделировать потерю доступа), затем проверить отображение статуса в списке согласно правилам продукта.

**Acceptance Scenarios**:

1. **Given** бот больше не состоит в чате или чат недоступен по данным от Telegram, **When** администратор открывает список чатов, **Then** он видит согласованное указание, что чат не следует считать активным получателем (или эквивалентное поведение, зафиксированное в продукте).

---

### User Story 4 — Регистрация webhook из настроек бота (Priority: P1)

Как администратор, я хочу из экрана **настроек бота** зарегистрировать webhook у Telegram для этого развёртывания, чтобы бот получал обновления и чаты попадали в учёт без ручной настройки вне админ-панели.

**Why this priority**: Без доставки обновлений на среду пользователя автоматическая регистрация чатов (US1) в проде не работает; это операционный предусловный сценарий.

**Independent Test**: На странице настроек бота выполнить действие регистрации webhook и убедиться, что Telegram принимает привязку (или отображается понятная ошибка), затем проверить, что обновления доходят до приложения штатным способом.

**Acceptance Scenarios**:

1. **Given** у администратора есть права на настройки бота, **When** он вводит **полный** URL webhook вручную и запускает регистрацию из настроек бота, **Then** система выполняет привязку этого URL у Telegram и сообщает об успехе или показывает понятное сообщение об ошибке без раскрытия секретов.
2. **Given** привязка уже выполнена, **When** администратор повторно вводит URL (тот же или новый) и регистрирует webhook, **Then** поведение остаётся предсказуемым: одна актуальная привязка, без дублирования конфликтующих сведений в интерфейсе.

---

### Edge Cases

- **Приватные и групповые чаты**: оба типа учитываются, если продукт их поддерживает; ограничения Telegram на получение метаданных отражаются запасными подписями.
- **Задержки Telegram**: чат может появиться в списке не мгновенно; админу показывается актуальное состояние после доставки событий.
- **Повторное добавление бота**: запись обновляется, а не множится.
- **Ошибки доставки событий**: временные сбои не должны безвозвратно «терять» чат; при восстановлении потока событий учёт догоняется.
- **Регистрация webhook**: неверный или недоступный URL, отсутствие HTTPS там, где это требуется от Telegram, отсутствие или неверный токен бота, отказ/лимит со стороны Telegram — администратор видит понятный итог, не раскрывающий внутренние секреты.
- **Конфликт URL**: если администратор указывает endpoint, не принадлежащий этому развёртыванию, продукт по возможности предупреждает до отправки или сообщает об ошибке после ответа Telegram.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST регистрировать каждый чат Telegram, для которого доступна информация о членстве бота или о входящей активности, достаточная для однозначной идентификации чата, так чтобы этот чат отображался в списке чатов администратора.
- **FR-002**: Система MUST обеспечивать отсутствие дублирующих записей для одного и того же логического чата Telegram.
- **FR-003**: Система MUST отображать администратору для каждого зарегистрированного чата стабильный идентификатор и отображаемое имя (или согласованный запасной вариант).
- **FR-004**: Система SHOULD обновлять сведения о чате (например, название, время последней активности) при поступлении новых сведений от канала связи с Telegram.
- **FR-005**: Система SHOULD отражать потерю членства бота в чате или недоступность чата в виде, понятном администратору при выборе получателей (например, признак «неактивен»), когда такие сведения доступны.
- **FR-006**: Система MUST предоставлять в **настройках бота** поле (или эквивалент) для **полного ввода вручную** URL webhook и действие, при котором выполняется **регистрация** этого URL у Telegram (привязка публичного HTTPS endpoint приёма обновлений к боту), без необходимости выходить из админ-панели к отдельным сторонним инструментам, не входящим в продукт. Автоматическое заполнение URL без действия администратора не требуется.
- **FR-007**: После попытки регистрации webhook система MUST показать администратору итог: успешная привязка или понятное сообщение об ошибке; в сообщениях для пользователя не допускается утечка секретов (токены, заголовки с секретом и т.п. в открытом виде).

### Key Entities *(include if feature involves data)*

- **Чат (Telegram)**: Логический чат во внешней сети; имеет стабильный внешний идентификатор; может иметь название, тип, признак активности с точки зрения бота.
- **Запись учёта чата**: То, что администратор видит в списке: связь внешнего чата с отображаемыми полями и статусом для операций (рассылки и т.д.).
- **Привязка webhook (логическая)**: Согласованное между продуктом и Telegram соответствие «бот → публичный endpoint приёма обновлений для этого развёртывания».

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В типичном сценарии «добавили бота в новый групповой чат» администратор видит этот чат в списке без ручного ввода данных (проверка сценарием из User Story 1).
- **SC-002**: Для одного и того же Telegram-чата в списке не появляется более одной логической строки с тем же идентификатором (проверка сценарием из User Story 1).
- **SC-003**: После сценария с удалением бота из чата (или эквивалента) администратор может отличить такой чат от активного с точки зрения доставки (проверка User Story 3).
- **SC-004**: Администратор может выполнить регистрацию webhook из настроек бота и получить однозначный результат (успех или понятная ошибка) без использования инструментов вне админ-панели (ручная проверка по User Story 4).

## Assumptions

- Администраторы уже аутентифицированы; права доступа к списку чатов и к настройкам бота согласованы с политикой продукта (детали ролей — вне объёма, если не введены отдельные роли).
- Поставщик Telegram доставляет боту события о членстве и/или сообщениях так, что продукт может извлечь идентификатор чата и при необходимости обновить метаданные, **при условии** что канал доставки обновлений (в т.ч. webhook) настроен на это развёртывание.
- «Разумное время» появления чата в списке — до нескольких минут после события при штатной работе канала связи; точные SLA не фиксируются в спецификации и могут быть уточнены при планировании.
- Ручной импорт списка чатов из файлов не требуется для базового сценария v1.
- Для регистрации webhook у Telegram требуется публично доступный **HTTPS** endpoint; **полный URL** вводит **вручную** администратор в настройках бота перед регистрацией (см. уточнение сессии 2026-04-01).
