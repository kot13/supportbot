# Feature Specification: Контекст и метаданные для неотвеченных вопросов

**Feature Branch**: `011-unanswered-context-metadata`  
**Created**: 2026-07-05  
**Status**: Draft  
**Input**: User description: «вместе с неотвеченным вопросом сохранять найденный контекст и все метаданные»

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Сохранение контекста при неотвеченном вопросе (Priority: P1)

Как администратор поддержки, я хочу, чтобы при каждом неотвеченном вопросе система сохраняла снимок того контекста, который бот использовал (или пытался использовать) для ответа, чтобы потом понять, почему ответ не был дан, без повторного воспроизведения ситуации вручную.

**Why this priority**: Без сохранённого контекста разбор неотвеченных вопросов сводится к догадкам; это основная ценность запроса.

**Independent Test**: Задать боту вопрос, на который он отвечает fallback-сообщением с причиной «недостаточно контекста», затем убедиться, что к записи неотвеченного вопроса привязан сохранённый снимок найденных фрагментов базы знаний и связанных метаданных.

**Acceptance Scenarios**:

1. **Given** бот обработал вопрос и пометил его как неотвеченный с причиной «релевантный контекст не найден» (`no_context`), **When** администратор открывает детали этой записи, **Then** он видит список фрагментов, которые были найдены при поиске, с их текстом, источником, оценкой релевантности и прочими сохранёнными метаданными.
2. **Given** бот обработал вопрос и пометил его как неотвеченный после ошибки генерации ответа (`openai_error`), **When** администратор открывает детали записи, **Then** он видит тот же снимок найденного контекста и метаданных, который был доступен боту на момент попытки ответа.
3. **Given** бот пометил вопрос как неотвеченный до этапа поиска по базе знаний (причины `no_knowledge_index` или `not_configured`), **When** администратор открывает детали записи, **Then** явно указано, что поиск контекста не выполнялся, и снимок контекста отсутствует или пуст.
4. **Given** бот успешно ответил на вопрос, **When** администратор просматривает неотвеченные вопросы, **Then** такой вопрос **не** появляется в списке и для него **не** создаётся снимок контекста.

---

### User Story 2 — Просмотр контекста в разделе неотвеченных вопросов (Priority: P2)

Как администратор, я хочу из раздела `/unanswered` быстро открыть сохранённый контекст и метаданные по конкретному вопросу, чтобы оценить качество поиска, выявить пробелы в базе знаний и решить, какой материал добавить или исправить.

**Why this priority**: Сохранение данных без удобного просмотра не даёт операционной пользы; просмотр — естественное продолжение P1.

**Independent Test**: Открыть `/unanswered`, развернуть или перейти к деталям записи с неотвеченным вопросом и проверить читаемое отображение всех сохранённых фрагментов и метаданных.

**Acceptance Scenarios**:

1. **Given** у неотвеченного вопроса есть сохранённый снимок контекста, **When** администратор просматривает эту запись в `/unanswered`, **Then** он может увидеть для каждого фрагмента: тип источника, путь/идентификатор источника, заголовок (если есть), текст фрагмента, оценку релевантности и дополнительные метаданные источника.
2. **Given** у неотвеченного вопроса при обработке использовалась недавняя переписка чата, **When** администратор открывает детали записи, **Then** он видит сохранённый диалоговый контекст (сообщения пользователя и бота), который учитывался при формировании ответа.
3. **Given** в списке `/unanswered` много записей, **When** администратор просматривает таблицу, **Then** основные поля (вопрос, причина, чат, пользователь, время) остаются компактными, а детальный контекст доступен по явному действию (разворот строки, модальное окно или отдельный экран деталей).

---

### User Story 3 — Аудит параметров обработки вопроса (Priority: P3)

Как администратор или владелец качества бота, я хочу видеть метаданные самого процесса обработки неотвеченного вопроса (причина отказа, момент фиксации, признак того, проходил ли поиск), чтобы отличать системные сбои от пробелов в документации и отслеживать тренды.

**Why this priority**: Дополняет контентный разбор; полезно для эксплуатации, но вторично относительно просмотра найденных фрагментов.

**Independent Test**: Создать несколько неотвеченных вопросов с разными причинами и убедиться, что в деталях каждой записи отображаются согласованные метаданные обработки.

**Acceptance Scenarios**:

1. **Given** вопрос помечен как неотвеченный, **When** администратор открывает детали, **Then** отображается причина неотвеченного вопроса в понятном для человека виде и время, когда вопрос был зафиксирован.
2. **Given** поиск по базе знаний был выполнен, **When** администратор смотрит метаданные обработки, **Then** видно, сколько фрагментов было возвращено и какая у лучшего фрагмента оценка релевантности (достаточна для сравнения «почти прошёл порог» vs «совсем мимо»).
3. **Given** поиск не выполнялся, **When** администратор смотрит метаданные, **Then** это явно отражено и не смешивается с пустым списком «ничего не найдено».

---

### Edge Cases

- Что происходит, если поиск вернул фрагменты, но все они ниже порога релевантности? Снимок сохраняется со всеми найденными фрагментами и их оценками; причина остаётся `no_context`.
- Что происходит при ошибке генерации ответа после успешного поиска? Сохраняется полный снимок найденного контекста; причина — `openai_error`.
- Что происходит, если у фрагмента базы знаний нет заголовка или дополнительных метаданных? Сохраняются доступные поля; отсутствующие отображаются как пустые, без потери остальных данных.
- Что происходит при повторной обработке того же сообщения (дубликат update)? Снимок контекста не перезаписывается повторно для уже зафиксированного неотвеченного вопроса.
- Что происходит с очень длинным текстом фрагментов? Все данные сохраняются полностью; в интерфейсе допускается сокращённый предпросмотр с возможностью раскрыть полный текст.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST при пометке пользовательского сообщения как неотвеченного сохранять вместе с ним неизменяемый снимок контекста, который был доступен боту на момент принятия решения об отказе от ответа.
- **FR-002**: Снимок контекста MUST включать все фрагменты, найденные при поиске по базе знаний: текст фрагмента, тип источника, путь/ключ источника, заголовок (если есть), оценку релевантности и все дополнительные метаданные фрагмента, имевшиеся в момент обработки.
- **FR-003**: Снимок MUST включать диалоговый контекст — недавние сообщения чата (роль и текст), которые учитывались при попытке ответа, если такой контекст использовался.
- **FR-004**: Система MUST сохранять метаданные обработки: причину неотвеченного вопроса, время фиксации, признак «поиск выполнялся / не выполнялся», количество найденных фрагментов и оценку релевантности лучшего фрагмента (если поиск выполнялся).
- **FR-005**: Снимок контекста MUST создаваться только для сообщений, помеченных как неотвеченные; успешные ответы бота не создают таких снимков.
- **FR-006**: Администратор MUST иметь возможность просмотреть сохранённый снимок и метаданные из раздела `/unanswered` без необходимости повторно задавать вопрос боту.
- **FR-007**: Отображение в списке `/unanswered` MUST сохранять текущую компактность таблицы; детальный контекст раскрывается по явному действию пользователя.
- **FR-008**: Для причин `no_knowledge_index` и `not_configured` система MUST явно фиксировать отсутствие этапа поиска и не подставлять пустой снимок за «нулевой результат поиска».
- **FR-009**: Сохранённые снимки MUST оставаться доступными для просмотра столько же, сколько хранится само сообщение в истории чата (без отдельного сокращённого срока хранения в рамках этой фичи).

### Key Entities

- **Неотвеченный вопрос**: Пользовательское сообщение в чате, помеченное причиной отказа бота от обоснованного ответа; связано с чатом, отправителем и временем.
- **Снимок контекста ответа**: Неизменяемая запись, привязанная к неотвеченному вопросу один-к-одному; содержит найденные фрагменты базы знаний, диалоговый контекст и метаданные обработки на момент попытки ответа.
- **Фрагмент в снимке**: Элемент найденного контекста с текстом, идентификацией источника, оценкой релевантности и произвольными метаданными источника на момент обработки.
- **Метаданные обработки**: Причина неотвеченного вопроса, факт и результат этапа поиска, агрегированные показатели (число фрагментов, лучшая релевантность), время фиксации.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% случаев, когда вопрос помечается как неотвеченный после выполнения поиска по базе знаний, администратор может открыть сохранённый снимок с полным списком найденных фрагментов и их метаданными без ручного воспроизведения вопроса.
- **SC-002**: Администратор может определить причину неотвеченного вопроса и оценить качество поиска (релевантность лучшего фрагмента, число кандидатов) менее чем за 2 минуты на одну запись через интерфейс `/unanswered`.
- **SC-003**: Для неотвеченных вопросов с причинами «индекс не готов» и «сервис не настроен» в 100% случаев в деталях явно различимы «поиск не выполнялся» и «поиск выполнен, но контекст недостаточен».
- **SC-004**: Количество обращений к разработчикам с просьбой «вручную воспроизвести, что видел бот» при разборе неотвеченных вопросов снижается за счёт самодостаточности записей в `/unanswered` (качественный критерий: администратор подтверждает, что деталей достаточно для решения о дополнении базы знаний).

## Assumptions

- Фича расширяет существующий поток неотвеченных вопросов (ветка `009-bot-qa-rag`); отдельный раздел или новая роль пользователей не требуются.
- «Найденный контекст» — это результат поиска по базе знаний плюс недавняя переписка чата, которые передавались в логику ответа; эмбеддинги и сырой запрос к внешнему API в снимок не включаются.
- «Все метаданные» — все поля фрагмента и процесса обработки, уже известные системе на момент ответа (источник, путь, заголовок, пользовательские метаданные фрагмента, distance/релевантность, причина, счётчики); секреты и ключи API не сохраняются.
- Просмотр снимка реализуется в рамках существующего раздела `/unanswered` (разворот, модальное окно или страница деталей) без отдельного экспорта в файлы в первой версии.
- Исторические неотвеченные вопросы, зафиксированные до внедрения фичи, могут остаться без снимка контекста; обратное заполнение не входит в scope.
