# Feature Specification: Автоматизация CI/CD через GitHub

**Feature Branch**: `010-github-cicd`  
**Created**: 2026-07-05  
**Status**: Draft  
**Input**: User description: «добавляем ci cd с помощью github.»

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Автоматическая проверка качества при изменениях кода (Priority: P1)

Как разработчик, я хочу, чтобы при каждом предложении изменений (pull request) и при слиянии в основную ветку автоматически запускались проверки качества, чтобы дефекты выявлялись до попадания кода в продакшен.

**Why this priority**: Без автоматических проверок команда полагается на ручной контроль; это основа CI и главный источник уверенности в стабильности репозитория.

**Independent Test**: Открыть pull request с заведомо ломающим изменением (например, падающий тест) и убедиться, что автоматическая проверка завершается неуспешно и сигнализирует об ошибке до слияния.

**Acceptance Scenarios**:

1. **Given** разработчик открывает или обновляет pull request, **When** изменения попадают в репозиторий, **Then** автоматически запускается набор проверок качества без ручного запуска.
2. **Given** проверки завершились, **When** разработчик смотрит статус pull request, **Then** он видит итог каждой проверки (успех или неуспех) и может перейти к деталям сбоя.
3. **Given** хотя бы одна обязательная проверка завершилась неуспешно, **When** мейнтейнер оценивает готовность к слиянию, **Then** pull request явно помечен как не готовый к merge (при включённой защите ветки — слияние заблокировано).
4. **Given** все обязательные проверки прошли успешно, **When** мейнтейнер смотрит pull request, **Then** статус отражает готовность к слиянию с точки зрения автоматических проверок.

---

### User Story 2 — Проверка собираемости приложения (Priority: P1)

Как мейнтейнер, я хочу, чтобы автоматическая проверка подтверждала, что приложение собирается из актуального кода, чтобы не обнаруживать ошибки сборки уже после слияния.

**Why this priority**: Сборка — минимальный барьер перед деплоем; без неё CI не защищает от регрессий, блокирующих релиз.

**Independent Test**: Внести в pull request изменение, ломающее сборку, и убедиться, что автоматическая проверка завершается неуспешно с понятным сообщением.

**Acceptance Scenarios**:

1. **Given** код в pull request или в основной ветке после merge, **When** запускается пайплайн проверок, **Then** выполняется этап проверки собираемости приложения.
2. **Given** код не собирается, **When** завершается этап сборки, **Then** весь пайплайн помечается как неуспешный.

---

### User Story 3 — Автоматический деплой после успешных проверок (Priority: P2)

Как мейнтейнер, я хочу, чтобы после успешного слияния в основную ветку приложение автоматически доставлялось в целевое окружение, чтобы сократить ручные шаги релиза и время от merge до доступности новой версии.

**Why this priority**: CD ускоряет поставку ценности, но зависит от работающего CI; без проверок автоматический деплой опасен.

**Independent Test**: Слить в основную ветку изменение, прошедшее все проверки, и убедиться, что новая версия появляется в целевом окружении без ручного деплоя.

**Acceptance Scenarios**:

1. **Given** изменения успешно слиты в основную ветку и все обязательные проверки на merge-коммите прошли, **When** срабатывает пайплайн доставки, **Then** приложение разворачивается в целевом окружении автоматически.
2. **Given** проверки на merge-коммите завершились неуспешно, **When** команда ожидает деплой, **Then** автоматический деплой не выполняется.
3. **Given** деплой запущен, **When** мейнтейнер смотрит историю запусков автоматизации, **Then** он видит статус деплоя (успех, неуспех, в процессе) и может найти причину сбоя.

---

### User Story 4 — Прозрачность и воспроизводимость для команды (Priority: P3)

Как участник команды, я хочу видеть единообразную историю запусков проверок и деплоев в интерфейсе репозитория, чтобы быстро разбирать инциденты и понимать, какая версия сейчас в окружении.

**Why this priority**: Улучшает операционную зрелость, но не блокирует базовый CI/CD.

**Independent Test**: Открыть вкладку автоматизации в репозитории и найти последние запуски с результатами, логами и привязкой к коммиту.

**Acceptance Scenarios**:

1. **Given** в репозитории были запуски проверок или деплоев, **When** участник команды открывает историю автоматизации, **Then** он видит список запусков с датой, веткой, коммитом и итоговым статусом.
2. **Given** запуск завершился неуспешно, **When** участник открывает детали запуска, **Then** он видит, на каком этапе произошла ошибка, без необходимости воспроизводить шаги локально.

---

### Edge Cases

- Что происходит при одновременном push в одну ветку несколькими коммитами — каждый коммит получает свой независимый запуск; устаревшие запуски не должны ошибочно помечать актуальный коммит как неуспешный.
- Как обрабатываются pull request из форков — проверки качества запускаются, но секреты деплоя и чувствительные данные не раскрываются.
- Что происходит при временной недоступности внешних сервисов (например, тестовой базы данных) — запуск помечается как неуспешный с явной причиной, а не «тихо» пропускается.
- Повторный запуск (re-run) после исправления — новый запуск отражает актуальное состояние кода без ручной настройки окружения.
- Отмена устаревшего деплоя при быстрых последовательных merge в основную ветку — в окружении должна оказаться последняя успешно прошедшая проверки версия.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST автоматически запускать проверки качества при открытии и обновлении pull request.
- **FR-002**: Система MUST автоматически запускать проверки качества при push в основную ветку.
- **FR-003**: Обязательный набор проверок MUST включать статический анализ кода (lint), автоматические тесты и проверку собираемости приложения.
- **FR-004**: Система MUST публиковать итоговый статус каждой проверки в контексте pull request (видимый участникам репозитория).
- **FR-005**: Система MUST предоставлять детальные логи каждого этапа проверки для диагностики сбоев.
- **FR-006**: При неуспехе любой обязательной проверки пайплайн MUST завершаться с общим статусом «неуспех».
- **FR-007**: Система MUST запускать автоматический деплой в целевое окружение только после успешного прохождения всех обязательных проверок на merge-коммите в основной ветке.
- **FR-008**: Секреты окружения (токены, пароли, ключи API) MUST храниться в защищённом хранилище платформы и не попадать в логи и артефакты.
- **FR-009**: Система MUST сохранять историю запусков проверок и деплоев с привязкой к коммиту и ветке.
- **FR-010**: Документация проекта MUST описывать, какие проверки выполняются, когда срабатывает деплой и какие секреты нужно настроить в репозитории.

### Key Entities

- **Запуск пайплайна (Pipeline Run)**: Единичное выполнение автоматизации; атрибуты: коммит, ветка, инициатор (push / pull request / ручной re-run), время начала и окончания, общий статус.
- **Этап проверки (Check Step)**: Отдельная операция внутри запуска (lint, тесты, сборка, деплой); атрибуты: имя, статус, длительность, ссылка на лог.
- **Статус в pull request (PR Check Status)**: Агрегированный результат обязательных этапов для конкретного коммита в PR; используется для решения о merge.
- **Целевое окружение (Deployment Target)**: Среда, куда доставляется приложение после успешного CD; атрибуты: имя окружения, версия (коммит), время последнего успешного деплоя.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% pull request в основной поток разработки получают результат автоматических проверок до merge без ручного запуска.
- **SC-002**: Разработчик видит итог всех обязательных проверок в интерфейсе pull request в течение 15 минут после push (при нормальной нагрузке на инфраструктуру).
- **SC-003**: Доля регрессий, связанных с несобираемым кодом или падающими тестами, попадающих в основную ветку, снижается до нуля при включённой защите ветки с обязательными статусами.
- **SC-004**: После успешного merge в основную ветку новая версия доступна в целевом окружении в течение 20 минут без ручного вмешательства (при нормальной нагрузке).
- **SC-005**: 100% неуспешных деплоев оставляют запись в истории автоматизации с указанием этапа и причины сбоя, доступной команде без доступа к серверу.
- **SC-006**: Время от обнаружения проблемы в PR до понимания причины (по логам автоматизации) — не более 5 минут для типовых сбоев тестов или сборки.

## Assumptions

- Репозиторий размещён на GitHub; автоматизация реализуется средствами GitHub (Actions и связанные возможности платформы).
- В проекте уже есть локальные команды проверки: lint, unit/integration-тесты и сборка; CI переиспользует тот же набор, что и разработчики локально.
- Интеграционные тесты требуют изолированной тестовой базы данных; в CI она поднимается автоматически на время запуска (детали — на этапе планирования).
- E2E-тесты (браузерные, с полным стеком приложения) **не входят** в обязательный набор проверок v1 из-за сложности окружения; могут быть добавлены отдельной итерацией.
- Целевое окружение для CD — **одно** production-подобное окружение; staging и blue-green деплой вне scope v1.
- Мейнтейнеры самостоятельно настраивают защиту основной ветки (required status checks) после появления первых успешных запусков.
- Секреты деплоя (доступ к серверу, переменные окружения продакшена) уже существуют или будут созданы мейнтейнерами в настройках репозитория; спецификация не описывает конкретный хостинг.
- Ручной деплой остаётся возможным как fallback, но не является основным сценарием после внедрения CD.
