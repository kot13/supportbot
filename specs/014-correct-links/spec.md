# Feature Specification: Формирование правильных ссылок в ответах бота

**Feature Branch**: `014-correct-links`  
**Created**: 2026-07-07  
**Status**: Draft  
**Input**: User description: «формирование правильных ссылок. 1) Ссылки на rest api должны вести на https://api.inappstory.ru/pub/v1#/ 2) Ссылки на статьи должны вести на адрес вида https://console.inappstory.ru/docs/{slug} — в файл resources.csv добавлена колонка slug 3) Ссылки на документацию должны вести на соответствующие разделы. Путь строится по пути к файлу в папке docs. Пример: ./docs/sdk-guides/android/how-to-get-started.md → https://docs.inappstory.ru/sdk-guides/android/how-to-get-started»

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Корректные ссылки на документацию SDK (Priority: P1)

Как пользователь Telegram, задающий вопрос о SDK InAppStory, я хочу получать в ответе бота кликабельные ссылки на актуальную документацию на `docs.inappstory.ru`, чтобы сразу перейти к нужному разделу без ручного поиска.

**Why this priority**: Документация SDK — основной источник ответов бота; неверные или битые ссылки напрямую снижают доверие к боту и увеличивают нагрузку на живую поддержку.

**Independent Test**: Задать боту вопрос, на который ответ опирается на фрагмент из `docs-master` (например, Android SDK «How to get started»), и проверить, что ссылка в ответе ведёт на `https://docs.inappstory.ru/sdk-guides/android/how-to-get-started`.

**Acceptance Scenarios**:

1. **Given** ответ бота основан на фрагменте документации с путём `sdk-guides/android/how-to-get-started.md`, **When** пользователь получает ответ со ссылкой на этот материал, **Then** ссылка имеет вид `https://docs.inappstory.ru/sdk-guides/android/how-to-get-started` (без суффикса `.md` и без префикса `docs/`).
2. **Given** ответ бота содержит ссылку на вложенный раздел документации (например, `glossarium/statistics/stories-widget-events.md`), **When** пользователь переходит по ссылке, **Then** открывается соответствующий URL `https://docs.inappstory.ru/glossarium/statistics/stories-widget-events`.
3. **Given** в ответе бота присутствует относительная ссылка на раздел документации (например, `/sdk-guides/android/how-to-get-started`), **When** сообщение доставляется пользователю в Telegram, **Then** ссылка разрешается в полный корректный URL на домене `docs.inappstory.ru`.

### User Story 2 — Корректные ссылки на статьи консоли (Priority: P1)

Как пользователь Telegram, задающий вопрос о работе в консоли InAppStory, я хочу получать ссылки на статьи на `console.inappstory.ru`, чтобы открыть нужную инструкцию в интерфейсе консоли.

**Why this priority**: Статьи консоли — второй по объёму источник знаний; без привязки к `slug` ссылки ведут на устаревшие или несуществующие адреса.

**Independent Test**: Задать боту вопрос, на который ответ опирается на статью из `resources.csv` со `slug=roles-management`, и проверить ссылку `https://console.inappstory.ru/docs/roles-management`.

**Acceptance Scenarios**:

1. **Given** ответ бота основан на статье из `resources.csv` со значением `slug=roles-management`, **When** пользователь получает ответ со ссылкой на эту статью, **Then** ссылка имеет вид `https://console.inappstory.ru/docs/roles-management`.
2. **Given** в базе знаний проиндексирована статья с заполненным полем `slug`, **When** бот ссылается на эту статью в ответе, **Then** в ссылке используется именно значение `slug`, а не внутренний идентификатор записи (`id`) или путь к файлу `resources.csv`.
3. **Given** ответ бота ссылается на несколько статей консоли, **When** пользователь просматривает ответ, **Then** каждая ссылка ведёт на свой уникальный URL вида `https://console.inappstory.ru/docs/{slug}`.

---

### User Story 3 — Корректные ссылки на публичное REST API (Priority: P1)

Как разработчик, спрашивающий бота о публичном REST API InAppStory, я хочу получать ссылки на официальную документацию API, чтобы быстро найти описание нужного эндпоинта.

**Why this priority**: Ошибочные ссылки на API (другой домен, устаревший портал) блокируют интеграцию и вынуждают искать документацию вручную.

**Independent Test**: Задать боту вопрос об эндпоинте публичного API (например, получение списка сторис) и проверить, что ссылка в ответе ведёт на портал `https://api.inappstory.ru/pub/v1#/`.

**Acceptance Scenarios**:

1. **Given** ответ бота основан на фрагменте из описания публичного REST API, **When** пользователь получает ответ со ссылкой на документацию API, **Then** ссылка ведёт на `https://api.inappstory.ru/pub/v1#/` (или на конкретный раздел внутри этого портала, если он однозначно определяется из источника).
2. **Given** ответ бота упоминает конкретную операцию API (например, `GET /v1/stories`), **When** в ответе присутствует ссылка на документацию этой операции, **Then** ссылка ведёт на портал `api.inappstory.ru`, а не на домены документации SDK или консоли.
3. **Given** пользователь получает ответ, сочетающий ссылки на SDK, консоль и API, **When** он переходит по каждой ссылке, **Then** каждая открывает материал соответствующего типа на правильном домене.

---

### User Story 4 — Единообразие ссылок при просмотре истории чата (Priority: P2)

Как администратор, просматривающий историю переписки в панели `/chats`, я хочу видеть в ответах бота те же корректные ссылки, что получил пользователь в Telegram, чтобы оценивать качество ответов и при необходимости проверять источники.

**Why this priority**: Консистентность между каналами доставки важна для контроля качества, но не блокирует основную ценность для конечного пользователя.

**Independent Test**: Открыть чат с ответом бота, содержащим ссылки, и сравнить URL с ожидаемыми шаблонами для трёх типов источников.

**Acceptance Scenarios**:

1. **Given** бот отправил ответ со ссылками в Telegram, **When** администратор открывает тот же чат в панели, **Then** ссылки в сохранённом ответе соответствуют правилам формирования URL для соответствующего типа источника.
2. **Given** ответ бота сохранён в истории чата, **When** администратор кликает по ссылке в панели, **Then** открывается тот же целевой ресурс, что и при переходе из Telegram.

---

### Edge Cases

- Что происходит, если у статьи в `resources.csv` отсутствует или пустое поле `slug`: бот не должен формировать ссылку с плейсхолдером или внутренним идентификатором; допускается ответ без ссылки на статью или с текстовым упоминанием названия без кликабельного URL.
- Как обрабатываются уже абсолютные ссылки в исходном контенте на устаревшие домены (например, `console.inappstory.com`, `docs.inappstory.com`): при формировании ответа бота ссылки на материалы InAppStory должны приводиться к актуальным доменам `.ru` и правилам данной спецификации.
- Как ведёт себя система при ссылке на корневой или индексный файл документации (например, `README.md` в подпапке): путь в URL строится по тому же правилу — относительный путь без `docs/` и без расширения `.md`.
- Что происходит, если один ответ ссылается на несколько источников разных типов: каждая ссылка формируется по правилу своего типа источника независимо от остальных.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST формировать ссылки на документацию SDK по шаблону `https://docs.inappstory.ru/{относительный-путь}`, где `{относительный-путь}` получается из пути файла в каталоге документации: убирается префикс `docs/` (если присутствует) и расширение `.md`.
- **FR-002**: Система MUST формировать ссылки на статьи консоли по шаблону `https://console.inappstory.ru/docs/{slug}`, где `{slug}` берётся из колонки `slug` соответствующей записи в `resources.csv`.
- **FR-003**: Система MUST формировать ссылки на публичное REST API, ведущие на портал документации API `https://api.inappstory.ru/pub/v1#/`.
- **FR-004**: Система MUST различать три типа источников знаний (документация SDK, статья консоли, спецификация API) и применять соответствующее правило формирования ссылки для каждого типа.
- **FR-005**: Система MUST обеспечивать, что ссылки в ответах бота в Telegram являются кликабельными и ведут на корректные внешние URL (не на внутренние пути приложения supportbot).
- **FR-006**: Система MUST сохранять в истории чата ответы бота с теми же URL, что были отправлены пользователю в Telegram.
- **FR-007**: При индексации статей из `resources.csv` система MUST учитывать колонку `slug` как атрибут источника, необходимый для построения ссылки на статью.
- **FR-008**: Система MUST NOT подставлять в ссылки на статьи консоли внутренние идентификаторы (`id` записи CSV) или технические пути вида `resources.csv#11`.
- **FR-009**: Система MUST NOT формировать ссылки на документацию SDK с доменом `docs.inappstory.com` или другими устаревшими доменами, если материал относится к актуальному порталу `docs.inappstory.ru`.
- **FR-010**: Если для статьи консоли `slug` отсутствует или пуст, система MUST NOT генерировать некорректную ссылку; допустимо опустить ссылку или указать только название статьи.

### Key Entities

- **Источник знаний (тип `sdk_doc`)**: фрагмент документации SDK; ключевой атрибут для ссылки — относительный путь к файлу в каталоге `docs` (без расширения).
- **Источник знаний (тип `console_article`)**: статья консоли из `resources.csv`; ключевые атрибуты — `slug`, `title`, `id` (только для внутренней идентификации, не для URL).
- **Источник знаний (тип `api_spec`)**: описание операции публичного REST API; ссылка ведёт на портал `https://api.inappstory.ru/pub/v1#/`.
- **Ответ бота**: сообщение с текстом и необязательными ссылками на материалы; сохраняется в истории чата и доставляется в Telegram.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В контрольной выборке из не менее 30 ответов бота (по 10 на каждый тип источника) не менее 95% ссылок соответствуют шаблонам URL, заданным в FR-001–FR-003.
- **SC-002**: 100% ссылок в контрольной выборке открывают ресурс на ожидаемом домене (`docs.inappstory.ru`, `console.inappstory.ru` или `api.inappstory.ru`) без редиректа на несвязанные домены.
- **SC-003**: При ручной проверке 10 ответов, ссылающихся на статьи консоли, ни одна ссылка не содержит внутренний идентификатор CSV или путь `resources.csv#`.
- **SC-004**: Время от клика по ссылке в Telegram до открытия целевой страницы документации не превышает обычного времени загрузки внешнего сайта (ссылки не проходят через промежуточные страницы supportbot).
- **SC-005**: Количество обращений в поддержку с жалобой «ссылка из бота не работает» снижается по сравнению с периодом до внедрения фичи (отслеживается качественно в течение первого месяца после релиза).

## Assumptions

- Колонка `slug` уже добавлена в `data/resources.csv` и заполнена для актуальных статей; задача фичи — использовать её при формировании ссылок, а не наполнять данные.
- Портал API `https://api.inappstory.ru/pub/v1#/` является единой точкой входа в документацию REST API; углублённые якоря на конкретные операции допустимы, если однозначно выводимы из источника, но базовый URL портала обязателен как минимум.
- Правило построения пути документации: файл `data/docs-master/docs/{path}.md` → URL `https://docs.inappstory.ru/{path}`; префикс `data/docs-master/docs/` не входит в публичный URL.
- Основной потребитель ссылок — пользователь Telegram; панель `/chats` и раздел неотвеченных вопросов наследуют те же правила для консистентности.
- Ссылки формируются на этапе подготовки или доставки ответа бота на основе метаданных проиндексированных источников; изменение правил не требует переобучения модели, но может потребовать переиндексации для подтягивания `slug` в метаданные чанков.
