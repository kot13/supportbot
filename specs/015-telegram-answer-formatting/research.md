# Research: Улучшение внешнего вида ответа в Telegram

**Branch**: `015-telegram-answer-formatting`  
**Date**: 2026-07-07

## Контекст

Текущий пайплайн ответа бота (`src/telegram/processMessage.ts`):

1. Модель генерирует ответ в Markdown (`src/rag/prompts.ts` — `SYSTEM_PROMPT` явно просит «Форматируй ответ в Markdown»).
2. Длинный текст режется функцией `splitLongTelegramText` по границам абзацев/строк.
3. Каждая часть конвертируется в Telegram HTML через `markdownToTelegramHtml`.
4. Сообщение отправляется с `parse_mode: HTML`.
5. В `chat_messages.content` сохраняется **исходный** Markdown модели (до санитизации).

Конвертер `markdownToTelegramHtml` поддерживает только подмножество Markdown: fenced code, inline code, ссылки, `**bold**`, `*italic*`. Заголовки, таблицы, горизонтальные разделители, зачёркивание, spoiler **не обрабатываются** — попадают в Telegram как сырой текст.

## R1: Стратегия обеспечения читаемого форматирования

**Decision**: Гибридный подход — **инструкции в промпте** + **детерминированная постобработка Markdown** перед конвертацией в HTML.

**Rationale**:

1. **Промпт** снижает частоту генерации таблиц и заголовков; задаёт стиль (жирные подзаголовки, списки, код).
2. **Постобработка** гарантирует FR-001/FR-003: даже если модель нарушила правила, пользователь не увидит `| ... |` или `## Заголовок`.
3. Конвертер HTML остаётся единой точкой доставки в Telegram; расширяется, но не дублируется.

**Alternatives considered**:

| Подход | Плюсы | Минусы | Вердикт |
|--------|-------|--------|---------|
| Только промпт | Минимальный diff | Модель непредсказуема в таблицах/заголовках | Отклонён |
| Только постобработка | Детерминизм | Модель продолжит тратить токены на бесполезную разметку | Недостаточно |
| MarkdownV2 вместо HTML | Нативный режим Telegram | Сложное экранирование, хрупкость | Отклонён |
| **Гибрид (выбран)** | Надёжность + лучшее качество генерации | Два слоя правил | Принят |

## R2: Поддерживаемые возможности Telegram (HTML parse_mode)

**Decision**: Целевой набор форматирования для ответов бота:

| Возможность | Markdown (ввод модели) | Telegram HTML (доставка) |
|-------------|------------------------|---------------------------|
| Жирный | `**текст**` | `<b>` |
| Курсив | `*текст*` | `<i>` |
| Инлайн-код | `` `код` `` | `<code>` |
| Блок кода | ` ```lang\n...\n``` ` | `<pre><code>` |
| Ссылка | `[текст](url)` | `<a href="...">` |
| Зачёркивание | `~~текст~~` | `<s>` |
| Подчёркивание | не генерировать активно; при наличии — `<u>` | `<u>` |
| Spoiler | не в v1 (редко в техподдержке) | `<tg-spoiler>` — опционально |
| Цитата | `> текст` | `<blockquote>` — опционально в v1 |

**Запрещённые конструкции** (удалять или преобразовывать):

- Заголовки ATX (`#`…`######`) → жирная строка без `#`
- Таблицы GFM (`| col |`) → текстовый список или строки через перенос
- Горизонтальные разделители (`---`, `***`, `___`) → пустая строка
- Сноски (`[^1]`) → удалить маркер
- Изображения `![alt](url)` → оставить `alt` как текст или удалить
- HTML-теги в тексте модели → экранировать (уже есть)

**Rationale**: Соответствует спецификации FR-001/FR-002; покрывает Bot API HTML без перехода на MarkdownV2.

**Alternatives considered**: Полный CommonMark-парсер (например `marked`) — новая зависимость, избыточен для узкой задачи.

## R3: Модуль постобработки Markdown

**Decision**: Новая функция `sanitizeMarkdownForTelegram(markdown: string): string` в `src/telegram/markdownToTelegramHtml.ts` (или соседний файл `telegramMarkdown.ts`), вызываемая **до** `markdownToTelegramHtml`.

Порядок этапов в `sendBotReply`:

```text
rawAnswer → sanitizeMarkdownForTelegram → splitLongTelegramText → markdownToTelegramHtml → sendMessage
```

Правила санитизации (детерминированные regex/построчная обработка):

1. Fenced code blocks — сохранить как есть (вынести в плейсхолдеры на время обработки остального текста).
2. Заголовки `#{1,6} ` → `**{title}**` + перевод строки.
3. Строки таблицы `|...|` → объединить ячейки через ` — ` или маркированный список для строк данных; строку-разделитель `|---|` удалить.
4. `^---+$` / `^\*\*\*+$` / `^___+$` → пустая строка.
5. Сноски `[^n]` и определения сносок — удалить.
6. Восстановить code blocks из плейсхолдеров.

**Rationale**: Изолированная чистая функция — легко покрыть unit-тестами (принцип IV).

## R4: Расширение `markdownToTelegramHtml`

**Decision**: Добавить конвертацию `~~strikethrough~~` → `<s>`; порядок замен сохранить: code blocks → inline code → links → bold → italic → strikethrough → escape.

Существующая логика ссылок (`resolveDocUrl`, нормализация `.com` → `.ru` из фичи `014-correct-links`) **не меняется**.

**Rationale**: Минимальное расширение существующего модуля; без новых npm-пакетов.

## R5: Разбиение длинных сообщений (FR-008)

**Decision**: Разбивать **после** санитизации, **до** HTML-конвертации; улучшить `splitLongTelegramText`:

- Не резать внутри fenced code block (если `cut` попадает в блок — сдвинуть границу к закрывающим ` ``` `).
- При неудаче — резать по `\n`, затем по `maxLen` (как сейчас).

**Rationale**: Разрез HTML посередине тега ломает `parse_mode`; разрез сырого Markdown проще контролировать.

**Alternatives considered**: Разбиение уже сконвертированного HTML — сложный парсер тегов; отклонён.

## R6: Что сохранять в `chat_messages.content` (FR-009)

**Decision**: Сохранять **санитизированный** Markdown (после `sanitizeMarkdownForTelegram`), а не сырой вывод модели.

**Rationale**: Администратор в `/chats` видит `whitespace-pre-wrap` без рендера Markdown; санитизированный текст совпадает с тем, что пользователь воспринимает в Telegram (без `#`, `|`, `---`). Сырой ответ модели теряется — приемлемо: цель — читаемость для QA.

**Alternatives considered**: Хранить оба варианта — требует миграции БД; вне scope.

## R7: Обновление `SYSTEM_PROMPT`

**Decision**: Заменить инструкцию «Форматируй ответ в Markdown» на явные правила Telegram-совместимого оформления:

- Разрешено: жирный, курсив, инлайн-код, блоки кода, ссылки, нумерованные и маркированные списки (plain text с `1.` / `•` / `-`).
- Запрещено: таблицы, заголовки `#`, горизонтальные линии `---`, изображения, сноски.
- Подзаголовки — **жирным** на отдельной строке.
- Сравнения — списком, не таблицей.

Правила ссылок из `014-correct-links` **сохраняются** без изменений.

**Rationale**: Превентивное снижение мусорной разметки; согласовано с постобработкой.

## R8: Панель `/chats`

**Decision**: Без изменений UI; достаточно санитизированного `content` в БД. Опционально в v2 — лёгкий рендер ссылок в админке.

**Rationale**: Спецификация P3 не требует визуального паритета с Telegram-клиентом.

## R9: Зависимости и миграции

**Decision**: Без новых npm-пакетов; без миграций БД; без переиндексации базы знаний.

**Rationale**: Фича затрагивает только генерацию/доставку ответа, не индекс.
