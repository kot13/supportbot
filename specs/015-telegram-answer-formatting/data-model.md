# Data model: Улучшение внешнего вида ответа в Telegram

**Branch**: `015-telegram-answer-formatting`  
**Date**: 2026-07-07

## Изменения схемы БД

**Миграции не требуются.** Используются существующие таблицы без изменения структуры.

## Сущности (логическая модель)

### Ответ бота (`BotAnswer`)

Текст, сгенерированный RAG-ассистентом до доставки пользователю.

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `rawMarkdown` | string | Исходный вывод модели (в памяти, не персистится после фичи) |
| `sanitizedMarkdown` | string | Markdown после `sanitizeMarkdownForTelegram` |
| `telegramHtml` | string | HTML для `parse_mode: HTML` (в памяти при отправке) |

**Жизненный цикл**:

```text
rawMarkdown (OpenAI)
    → sanitizeMarkdownForTelegram
    → sanitizedMarkdown
    → splitLongTelegramText (0..N частей)
    → markdownToTelegramHtml (на часть)
    → telegramHtml → Telegram API
```

`sanitizedMarkdown` (полный текст, до разбиения) → `chat_messages.content`.

### Правила оформления (`FormattingRules`)

Набор допустимых и запрещённых конструкций; реализован как константы/функции в `src/telegram/`, не в БД.

| Категория | Допустимо | Запрещено / преобразуется |
|-----------|-----------|---------------------------|
| Акценты | `**bold**`, `*italic*`, `` `code` ``, `~~strike~~` | — |
| Структура | списки `1.` / `-` / `•`, абзацы, пустые строки | `# заголовки`, `---` |
| Данные | текст, ссылки | GFM-таблицы `\|...\|` |
| Медиа | — | `![image](url)` |
| Ссылки | `[label](url)` | — (URL по правилам `014-correct-links`) |

### Сообщение чата (`chat_messages`) — семантика поля `content`

| Поле | До фичи | После фичи |
|------|---------|------------|
| `content` (role=bot) | Сырой Markdown модели | Санитизированный Markdown (Telegram-совместимый) |
| `content` (role=user) | Без изменений | Без изменений |

**Правило FR-009**: значение `content` для ответов бота MUST быть читаемым без сырой неподдерживаемой разметки.

### Части длинного сообщения (`TelegramMessagePart`)

При превышении лимита 4096 символов:

| Атрибут | Описание |
|---------|----------|
| `index` | Порядковый номер части (0-based) |
| `markdownSlice` | Фрагмент `sanitizedMarkdown` |
| `htmlSlice` | Результат `markdownToTelegramHtml(markdownSlice)` |

**Инвариант**: граница разреза MUST NOT попадать внутрь fenced code block.

## Взаимодействие с существующими сущностями

| Сущность | Изменение |
|----------|-----------|
| `knowledge_chunks` | Нет |
| `unanswered_context_snapshots` | Нет (хранит контекст поиска, не формат ответа) |
| `bot_settings` | Нет |
| Telegram `sendMessage` | Без изменения контракта API; тот же `parse_mode: HTML` |

## Валидация и ограничения

1. Длина одной части после HTML-конвертации ≤ 4096 символов (лимит Telegram); при превышении — повторное разбиение на уровне markdown (редкий случай из-за экранирования).
2. HTML MUST быть валидным для Telegram Bot API: только разрешённые теги, экранированные `<`, `>`, `&` вне тегов.
3. При ошибке отправки HTML — fallback на plain text части (существующее поведение в `sendBotReply`).

## Состояния ошибок

| Ситуация | Поведение |
|----------|-----------|
| HTML parse error от Telegram | Повторная отправка той же части без `parse_mode` |
| Пустой ответ после санитизации | Не должно возникать; если пусто — не отправлять / fallback на `OPENAI_ERROR_MESSAGE` |
| Модель вернула только таблицу | Санитизация преобразует в текстовый список |
