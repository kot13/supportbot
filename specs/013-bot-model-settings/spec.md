# Feature Specification: Выбор моделей эмбеддинга и ответов в настройках бота

**Feature Branch**: `013-bot-model-settings`  
**Created**: 2026-07-05  
**Status**: Draft  
**Input**: User description: «добавить возможность выбирать модель для эмбеддинга и ответов в настройках бота. Эмбеддинги: text-embedding-3-small (по-умолчанию), text-embedding-3-large. Ответы бота: gpt-4.1 (по-умолчанию), gpt-5.5»

## Clarifications

### Session 2026-07-05

- Q: Как администратор запускает переиндексацию после смены модели эмбеддингов? → A: И кнопка на экране настроек бота, и CLI-индексация; оба пути MUST использовать сохранённую в настройках модель эмбеддингов.
- Q: Показывать ли на экране настроек актуальность индекса относительно текущей модели эмбеддингов? → A: Индикатор «индекс устарел», если модель эмбеддингов в настройках отличается от модели последней успешной переиндексации, плюс время последней успешной переиндексации.
- Q: Должен ли экран настроек автоматически обновлять статус переиндексации до завершения? → A: Да — статус на экране настроек автоматически обновляется до успешного завершения или ошибки.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Выбор модели для ответов бота (Priority: P1)

Как администратор, я хочу в настройках бота выбрать, какая модель будет генерировать ответы на вопросы пользователей, чтобы балансировать между качеством ответов и стоимостью обслуживания.

**Why this priority**: Модель ответов напрямую влияет на качество поддержки и воспринимаемую ценность бота; это основной рычаг настройки поведения бота без изменения кода.

**Independent Test**: Открыть настройки бота, выбрать альтернативную модель ответов, сохранить, задать боту вопрос в Telegram — убедиться, что ответ сформирован с использованием выбранной модели (по журналу операций или метаданным ответа, доступным администратору).

**Acceptance Scenarios**:

1. **Given** администратор авторизован в панели и открыл настройки бота, **When** он просматривает раздел выбора модели ответов, **Then** доступны ровно два варианта: `gpt-4.1` и `gpt-5.5`, с понятными подписями.
2. **Given** настройки ещё не менялись, **When** администратор открывает настройки бота, **Then** выбрана модель ответов `gpt-4.1` по умолчанию.
3. **Given** администратор выбрал `gpt-5.5` и сохранил настройки, **When** пользователь задаёт боту вопрос, требующий генерации ответа, **Then** ответ формируется с использованием модели `gpt-5.5`.
4. **Given** администратор изменил модель ответов, **When** сохранение завершилось успешно, **Then** новый выбор сохраняется и отображается при следующем открытии настроек.
5. **Given** администратор изменил модель ответов, **When** сохранение завершилось успешно, **Then** все **новые** ответы бота используют обновлённую модель; уже отправленные ответы не пересчитываются.

### User Story 2 — Выбор модели для эмбеддингов (Priority: P1)

Как администратор, я хочу в настройках бота выбрать модель для построения векторных представлений базы знаний, чтобы управлять точностью поиска по документации и связанными с этим затратами.

**Why this priority**: Модель эмбеддингов определяет качество поиска релевантного контекста; без корректного выбора ответы бота могут опираться на нерелевантные фрагменты документации.

**Independent Test**: Выбрать альтернативную модель эмбеддингов, выполнить переиндексацию базы знаний, задать боту вопрос — убедиться, что поиск и ответы работают с новой моделью.

**Acceptance Scenarios**:

1. **Given** администратор авторизован в панели и открыл настройки бота, **When** он просматривает раздел выбора модели эмбеддингов, **Then** доступны ровно два варианта: `text-embedding-3-small` и `text-embedding-3-large`, с понятными подписями.
2. **Given** настройки ещё не менялись, **When** администратор открывает настройки бота, **Then** выбрана модель эмбеддингов `text-embedding-3-small` по умолчанию.
3. **Given** администратор выбрал `text-embedding-3-large` и сохранил настройки, **When** выполняется индексация базы знаний, **Then** векторные представления строятся с использованием `text-embedding-3-large`.
4. **Given** администратор выбрал `text-embedding-3-large` и сохранил настройки, **When** бот ищет релевантный контекст для ответа, **Then** запрос пользователя векторизуется той же моделью эмбеддингов, что указана в настройках.
5. **Given** администратор изменил модель эмбеддингов, **When** сохранение завершилось успешно, **Then** система явно предупреждает, что для корректной работы поиска требуется переиндексация базы знаний.

### User Story 4 — Переиндексация базы знаний после смены модели эмбеддингов (Priority: P1)

Как администратор, я хочу запустить переиндексацию базы знаний прямо из настроек бота (или через CLI), чтобы после смены модели эмбеддингов поиск снова работал корректно без обращения к разработчикам.

**Why this priority**: Смена модели эмбеддингов бесполезна без переиндексации; без явного действия администратор рискует оставить бота в нерабочем состоянии поиска.

**Independent Test**: Сменить модель эмбеддингов, нажать «Переиндексировать базу знаний» на экране настроек, дождаться завершения — задать боту вопрос и убедиться, что ответ опирается на релевантный контекст. Альтернативно: выполнить CLI-индексацию и проверить тот же результат.

**Acceptance Scenarios**:

1. **Given** администратор на экране настроек бота, **When** он нажимает действие переиндексации базы знаний, **Then** запускается полная переиндексация с использованием текущей сохранённой модели эмбеддингов.
2. **Given** администратор выполняет CLI-индексацию (`rag:index`), **When** процесс запускается, **Then** векторные представления строятся с использованием модели эмбеддингов из сохранённых настроек бота (а не захардкоженного значения).
3. **Given** переиндексация уже выполняется, **When** администратор пытается запустить её повторно из UI, **Then** система сообщает, что индексация уже идёт, и не запускает второй параллельный процесс.
4. **Given** переиндексация запущена из UI, **When** администратор остаётся на экране настроек, **Then** статус «выполняется» автоматически обновляется до «готово» или «ошибка» без ручной перезагрузки страницы.
5. **Given** переиндексация завершилась успешно, **When** администратор остаётся на экране настроек, **Then** он видит подтверждение успешного завершения, актуальный статус «готово», обновлённое время последней успешной переиндексации и снятый индикатор «индекс устарел» (если модель эмбеддингов совпадает).
6. **Given** переиндексация завершилась с ошибкой, **When** администратор смотрит экран настроек, **Then** он видит понятное сообщение об ошибке без технических секретов.

### User Story 3 — Единый экран настроек бота (Priority: P2)

Как администратор, я хочу видеть и менять выбор моделей на том же экране, где настраиваются имя и токен бота, чтобы все параметры бота были собраны в одном месте.

**Why this priority**: Удобство эксплуатации; не блокирует саму возможность выбора моделей, но снижает риск пропустить настройку.

**Independent Test**: Открыть страницу настроек бота — убедиться, что поля имени, токена и оба селектора моделей отображаются на одном экране и сохраняются одной операцией «Сохранить».

**Acceptance Scenarios**:

1. **Given** администратор на странице настроек бота, **When** он просматривает форму, **Then** рядом с существующими полями (имя, токен) отображаются элементы выбора модели ответов, модели эмбеддингов, блок статуса индекса и действие переиндексации базы знаний.
2. **Given** текущая модель эмбеддингов в настройках отличается от модели, с которой выполнялась последняя успешная переиндексация (или переиндексация ещё ни разу не выполнялась), **When** администратор открывает настройки бота, **Then** он видит индикатор «индекс устарел» (или эквивалентное понятное состояние).
3. **Given** хотя бы одна успешная переиндексация выполнялась ранее, **When** администратор открывает настройки бота, **Then** он видит время последней успешной переиндексации.
4. **Given** последняя успешная переиндексация выполнялась с той же моделью эмбеддингов, что сейчас в настройках, **When** администратор открывает настройки бота, **Then** индикатор «индекс устарел» не отображается.
5. **Given** администратор изменил только модели (без смены токена), **When** он сохраняет форму, **Then** настройки моделей обновляются без требования повторно вводить токен.
6. **Given** сохранение настроек завершилось успешно, **When** администратор остаётся на странице, **Then** он видит подтверждение успешного сохранения.

---

### Edge Cases

- **Индекс ещё ни разу не строился**: индикатор «индекс устарел» (или «не проиндексировано») отображается; время последней успешной переиндексации не показывается.
- **Смена модели эмбеддингов без переиндексации**: поиск по базе знаний может давать нерелевантные или пустые результаты; система MUST предупреждать администратора при сохранении новой модели эмбеддингов, MUST показывать индикатор «индекс устарел» на экране настроек и MUST не выдавать уверенные ответы на основе несовместимого индекса (поведение согласуется с существующей логикой «нет контекста» / fallback).
- **Параллельный запуск переиндексации**: повторный запуск из UI или CLI, пока индексация уже идёт, отклоняется с понятным сообщением; активный процесс не прерывается.
- **Длительная переиндексация**: пока процесс идёт, экран настроек показывает статус «выполняется» и автоматически обновляет его до завершения или ошибки; кнопка повторного запуска недоступна.
- **CLI при отсутствии настроек модели**: CLI-индексация использует значение по умолчанию (`text-embedding-3-small`), если настройки ещё не заданы.
- **Недоступность выбранной модели**: если внешний сервис генерации временно не принимает запросы к выбранной модели, бот отвечает понятным fallback-сообщением без технических деталей; вопрос может попасть в список неотвеченных.
- **Отсутствие API-ключа**: при ненастроенном ключе доступа к сервису генерации выбор модели сохраняется, но ответы бота не генерируются — существующее поведение `not_configured` сохраняется.
- **Параллельные запросы во время смены модели**: запросы, принятые до сохранения новых настроек, обрабатываются по прежним моделям; запросы после сохранения — по новым.
- **Попытка указать модель вне разрешённого списка**: система отклоняет недопустимое значение с понятным сообщением об ошибке; сохранённые ранее корректные значения не затираются.
- **Первый запуск / миграция существующих установок**: для систем, где настройки моделей ещё не заданы, применяются значения по умолчанию (`text-embedding-3-small`, `gpt-4.1`) без действий администратора.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять администратору в настройках бота выбор модели для генерации ответов из фиксированного списка: `gpt-4.1`, `gpt-5.5`.
- **FR-002**: Система MUST предоставлять администратору в настройках бота выбор модели для эмбеддингов из фиксированного списка: `text-embedding-3-small`, `text-embedding-3-large`.
- **FR-003**: Значения по умолчанию MUST быть: модель ответов — `gpt-4.1`, модель эмбеддингов — `text-embedding-3-small`.
- **FR-004**: Система MUST сохранять выбранные модели вместе с прочими настройками бота и восстанавливать их при следующем открытии настроек.
- **FR-005**: Все новые ответы бота MUST формироваться с использованием сохранённой модели ответов.
- **FR-006**: Индексация базы знаний и векторизация пользовательских запросов MUST использовать сохранённую модель эмбеддингов.
- **FR-007**: При сохранении смены модели эмбеддингов система MUST показывать предупреждение о необходимости переиндексации базы знаний.
- **FR-008**: Система MUST отклонять значения моделей, не входящие в разрешённые списки, с понятным сообщением об ошибке валидации.
- **FR-009**: Элементы выбора моделей MUST располагаться на существующем экране настроек бота и сохраняться той же операцией, что и остальные настройки бота.
- **FR-010**: Изменение модели ответов MUST NOT требовать повторного ввода токена бота, если токен уже сохранён.
- **FR-011**: Доступ к изменению моделей MUST быть ограничен теми же правами, что и доступ к остальным настройкам бота (только авторизованный администратор панели).
- **FR-012**: На экране настроек бота MUST быть доступно действие для запуска полной переиндексации базы знаний (тот же смысл, что CLI `rag:index`).
- **FR-013**: CLI-индексация (`rag:index`) MUST использовать модель эмбеддингов из сохранённых настроек бота.
- **FR-014**: UI-переиндексация MUST использовать ту же логику полной переиндексации, что и CLI; оба пути MUST применять текущую сохранённую модель эмбеддингов.
- **FR-015**: При попытке запустить переиндексацию, пока она уже выполняется, система MUST отклонять запрос с понятным сообщением (без параллельных запусков).
- **FR-016**: На экране настроек бота MUST отображаться статус актуальности индекса: индикатор «индекс устарел», если текущая модель эмбеддингов отличается от модели последней успешной переиндексации или успешная переиндексация ещё не выполнялась.
- **FR-017**: На экране настроек бота MUST отображаться время последней успешной переиндексации, если такая переиндексация выполнялась хотя бы один раз.
- **FR-018**: Система MUST сохранять модель эмбеддингов, использованную при каждой успешной переиндексации, чтобы сравнивать её с текущими настройками.
- **FR-019**: Пока переиндексация выполняется, экран настроек бота MUST автоматически обновлять отображаемый статус до перехода в «готово» или «ошибка» без ручной перезагрузки страницы.
- **FR-020**: Во время выполнения переиндексации действие повторного запуска MUST быть недоступно.

### Key Entities

- **Настройки моделей бота**: пара сохранённых значений — модель для эмбеддингов и модель для ответов; относится к общим настройкам бота; используется при индексации базы знаний, поиске контекста и генерации ответов.
- **Разрешённый каталог моделей**: фиксированные перечни допустимых моделей для каждого назначения (эмбеддинги / ответы), задаваемые продуктом; не расширяется администратором произвольным вводом.
- **Запуск переиндексации**: действие администратора (из UI настроек бота или CLI), инициирующее полную пересборку векторного индекса базы знаний с текущей моделью эмбеддингов.
- **Статус индекса базы знаний**: отображаемое состояние — актуален / устарел / выполняется / ошибка; включает время последней успешной переиндексации и модель эмбеддингов, с которой она выполнялась.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Администратор может просмотреть текущие модели и сохранить новый выбор менее чем за 1 минуту без обращения к разработчикам.
- **SC-002**: В 100% проверенных сценариев после сохранения настроек новые ответы бота используют выбранную модель ответов.
- **SC-003**: В 100% проверенных сценариев после сохранения настроек и переиндексации поиск по базе знаний использует выбранную модель эмбеддингов.
- **SC-004**: При смене модели эмбеддингов 100% администраторов видят предупреждение о необходимости переиндексации до или сразу после сохранения.
- **SC-005**: После перезагрузки страницы или повторного входа в панель сохранённые значения моделей отображаются без расхождений с последним успешным сохранением.
- **SC-006**: Администратор может запустить переиндексацию из настроек бота менее чем за 30 секунд (без SSH/CLI); после успешного завершения бот отвечает на вопросы с использованием новой модели эмбеддингов.
- **SC-007**: В 100% проверенных сценариев после смены модели эмбеддингов без переиндексации администратор видит индикатор «индекс устарел» на экране настроек; после успешной переиндексации индикатор исчезает.
- **SC-008**: При запуске переиндексации из UI администратор видит финальный статус (успех или ошибка) на том же экране без ручной перезагрузки страницы.

## Assumptions

- Доступ к обеим моделям ответов (`gpt-4.1`, `gpt-5.5`) и обеим моделям эмбеддингов (`text-embedding-3-small`, `text-embedding-3-large`) обеспечивается существующим ключом доступа к сервису генерации, настроенным в окружении развёртывания.
- Расширение списка моделей произвольным вводом администратором вне объёма этой фичи; поддерживаются только четыре указанные модели.
- Переиндексация при смене модели эмбеддингов инициируется администратором вручную — через кнопку на экране настроек бота или CLI `rag:index`; автоматический запуск при сохранении настроек не требуется в v1.
- Существующий API переиндексации (`POST /api/knowledge/reindex`) может быть переиспользован для UI-кнопки на экране настроек.
- Для сравнения актуальности индекса система хранит модель эмбеддингов, использованную при последней успешной переиндексации (в метаданных запуска индексации или эквивалентном месте).
- Размерность векторов и совместимость индекса при смене модели эмбеддингов — ответственность этапа планирования/реализации; на уровне спецификации зафиксировано требование предупреждать администратора и не полагаться на устаревший индекс.
- Поведение бота при отсутствии контекста, ошибках сервиса и мониторинг неотвеченных вопросов наследуется из существующей фичи ответов бота (009) без изменения пользовательских сообщений.

## Dependencies

- Существующий экран и API настроек бота (имя, токен).
- Существующий пайплайн ответов бота на вопросы, CLI `rag:index` и API переиндексации базы знаний.
- Настроенный ключ доступа к внешнему сервису генерации ответов и эмбеддингов.
