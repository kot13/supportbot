# Research: Видео в рассылке (008)

## 1. Telegram: видео с подписью и несколько видео

**Decision**:

- Одно видео: метод **`sendVideo`** с полями `chat_id`, `video` (multipart), `caption`, при необходимости `parse_mode`.
- Несколько медиа (2–10): **`sendMediaGroup`** с массивом `media`, где элементы типа **`video`** с `attach://…` для каждого файла; подпись только на **первом** элементе (как сейчас для альбома фото в `sendMediaGroup.ts`).

**Rationale**: Совпадает с уже реализованным паттерном для изображений; документированные лимиты Telegram: до 10 элементов в группе; caption до 1024 символов для медиа с подписью.

**Alternatives considered**:

- Отправка только одного видео в v1 — проще, но хуже паритет с лимитом «до 10» для фото.
- Смешение фото и видео в одной группе — допускается API Telegram, но усложняет UI и тесты; **отложено** за рамками v1 (взаимоисключение с изображениями в плане).

## 2. Лимиты размера и формата

**Decision**:

- На сервере: жёсткий **max size per file** (например **50 MiB** как ориентир под Bot API; уточнить при реализации по актуальной документации Telegram).
- MIME: минимум **`video/mp4`**, опционально **`video/quicktime`** (MOV) и др. по белому списку.
- Отклонять пустой `type` от браузера по расширению как вторичный сигнал (как договорённость проекта).

**Rationale**: FR-003/FR-004 требуют отказа до массовой отправки; единый белый список проще сопровождать.

**Alternatives considered**:

- Полагаться только на `file.type` — недостаточно для надёжности; комбинация размер + MIME + расширение.

## 3. Взаимоисключение «изображения / видео»

**Decision**: В **v1** при выборе видео очищать список изображений и наоборот; показывать короткую подсказку в UI.

**Rationale**: Выполняет FR-006 и допущения спеки без реализации смешанных медиагрупп в первой поставке.

**Alternatives considered**:

- Разрешить смешение и строить смешанный `sendMediaGroup` — отложено.

## 4. База данных

**Decision**: Использовать **`broadcast_attachments`** для строк видео (`mime_type` вида `video/...`); `telegram_file_id` заполнять после успешной отправки, если текущая логика это делает для изображений.

**Rationale**: Таблица уже универсальна по MIME; отдельная миграция **не обязательна** для MVP.

**Alternatives considered**:

- Колонка `media_kind` — улучшает запросы и отчёты; опциональная миграция во второй итерации.
